# Finder System – Assignment Report

---

## 1. Introduction

### 1.1 Overview

This report describes the design, implementation, and evaluation of a **Finder System** – a distributed application built in Java that allows networked clients to discover and connect to each other through a central intermediary called the **Finder Server**.

The system was developed as part of a distributed-systems assignment to demonstrate key concepts including client–server communication, TCP socket programming, multi-threading, and graphical user interfaces.

### 1.2 Scope

The scope of the project covers:

- A central **Finder Server** that maintains a live registry of client names, network locations (host and port), and unique connection codes.
- Multiple **Client** instances that can register themselves with the server, look up other registered clients by name, and connect directly to a remote peer after obtaining its connection details through the Finder.
- A **GUI** for both the Server and each Client that lets a user see system activity and trigger actions without any command-line interaction.
- All communication is over **TCP sockets** using Java's `DataInputStream` / `DataOutputStream` streams.

The project is intentionally kept simple and self-contained: it uses only the Java SE standard library and the NetBeans GUI framework (Swing), with no external dependencies or databases.

## 2. Design

### 2.1 Architecture

The system follows a **three-phase client-server architecture**:

```
Phase 1 – Registration
  Client A  ──────REGISTER──────►  Finder Server
            ◄──── Code: 1000 ──────

Phase 2 – Discovery
  Client B  ──────FIND Alice──────►  Finder Server
            ◄── localhost:6001:1000──

Phase 3 – Peer Connection (direct)
  Client B  ──────code "1000"──────►  Client A (listening on 6001)
            ◄──── "Connected" ───────
```

The Finder Server is **not** involved in Phase 3; it only acts as a directory. The unique code ensures that a client cannot connect to a peer without first going through the Finder (because only the Finder hands out the code).

### 2.3 Protocol

All messages are plain UTF strings exchanged with `DataOutputStream.writeUTF()` / `DataInputStream.readUTF()`.

| Direction | Message format | Server reply |
|---|---|---|
| Client → Server | `REGISTER <name> <host> <port>` | `Registered: <name> Code: <code>` or `Already registered: <name>` |
| Client → Server | `FIND <name>` | `<host>:<port>:<code>` or `Not found` |
| Client → Server | `UNREGISTER <name>` | `Unregistered: <name>` or `Not found: <name>` |
| Connecting client → Target client | `<code>` (raw string) | `Connected` or `Invalid code` |

### 2.4 Data Model

The Finder Server stores each registered client as a four-element `String[]` array inside a static `ArrayList`:

```
index 0 → name   (e.g. "Alice")
index 1 → host   (e.g. "localhost")
index 2 → port   (e.g. "6001")
index 3 → code   (e.g. "1000")
```

Unique codes are generated by an `AtomicInteger` counter (thread-safe, starts at 1000) incremented on every successful registration.

### 2.5 Threading Model

| Component | Thread |
|---|---|
| `FinderServerForm` – accept loop | Dedicated background thread (started when "Start" clicked) |
| `FinderServer` – each client request | One new `Thread` per accepted connection |
| `ClientForm` – peer listener | One background thread started after successful registration |
| `ClientForm` – each incoming peer | One new `Thread` per accepted peer connection |

## 3. Implementation

### 3.1 Server Side

#### `FinderServer.java` (Worker class)

`FinderServer` implements `Runnable`. Each time a client connects to the server port, `FinderServerForm` wraps the accepted `Socket` in a new `FinderServer` instance and starts it on its own thread. This means multiple clients can interact with the server simultaneously without blocking each other.

**Key fields:**

```java
static ArrayList<String[]> clients         // shared registry across all threads
static AtomicInteger codeCounter           // thread-safe code generator
Socket cSocket                             // the client connection this worker handles
```

**`run()` – command dispatcher:**

The method reads a single UTF command string and branches on the command keyword:

- **`REGISTER name host port`**  
  Checks if the name is already in the list. If not, it generates the next code (`codeCounter.getAndIncrement()`), adds `{name, host, port, code}` to the list, and replies with `"Registered: <name> Code: <code>"`. The GUI log is updated via `FinderServerForm.log(...)`.

- **`FIND name`**  
  Iterates the list. If the name is found it replies with `"host:port:code"` (the colon-delimited triple the connecting client needs). If not found it replies `"Not found"`.

- **`UNREGISTER name`**  
  Iterates the list and removes the entry if found, or replies `"Not found: name"`.

After handling any command the worker closes its streams and socket.

#### `FinderServerForm.java` (GUI class)

Provides the **Finder Server GUI** – a single `JFrame` containing:

| Component | Purpose |
|---|---|
| Port field (`jTextField1`, default `5000`) | Lets the operator choose the listening port |
| Start button (`jButton1`) | Creates the `ServerSocket` and starts the accept loop |
| Activity log (`jTextArea1`) | Append-only text area showing every registration, lookup and unregistration event |

The **Start button handler** creates the `ServerSocket`, disables itself (so it cannot be clicked twice), and starts a background thread that loops calling `sSocket.accept()`. Each accepted connection is handed off to a new `FinderServer` thread.

The static `log(String msg)` method allows `FinderServer` worker threads to append messages to the GUI text area safely.

### 3.2 Client Side

#### `Client.java` (Helper class)

Contains four **static utility methods** that each open a fresh TCP connection, send one command, read one reply, close the connection, and return the reply string. This keeps the networking logic completely separate from the GUI.

| Method | Sends | Returns |
|---|---|---|
| `register(host, port, name, myPort)` | `REGISTER name localhost myPort` | Server reply (includes `Code:` on success) |
| `unregister(host, port, name)` | `UNREGISTER name` | Server reply |
| `find(host, port, name)` | `FIND name` | `host:port:code` or `Not found` |
| `connectToPeer(host, port, code)` | `code` (string) | `Connected` or `Invalid code` |

#### `ClientForm.java` (GUI class)

Provides the **Client GUI** – a `JFrame` containing:

| Component | Purpose |
|---|---|
| Name field (`jTextField1`, default `Client1`) | This client's display name |
| My Port field (`jTextField2`, default `6001`) | The port this client listens on for peer connections |
| Server Host / Server Port (`jTextField3`, `jTextField4`) | Address of the Finder Server |
| Register button (`jButton1`) | Registers with the server and starts the peer listener |
| Unregister button (`jButton2`) | Removes this client from the server registry |
| Find Name field (`jTextField5`) | Target client name to look up or connect to |
| Find button (`jButton3`) | Looks up the named client and prints `host:port:code` |
| Connect button (`jButton4`) | Looks up the named client then calls `connectToPeer` directly |
| Activity log (`jTextArea1`) | Append-only output showing all events |

**Instance fields:**

```java
String  myCode    = "";     // code assigned by server on registration
boolean listening = false;  // prevents starting the listener twice
```

**Register button handler (`jButton1ActionPerformed`):**

1. Calls `Client.register(...)` and prints the result.
2. Parses the code out of `"Registered: name Code: XXXX"` and stores it in `myCode`.
3. If `listening == false`, starts a background `ServerSocket` thread on `myPort`:
   - Loops calling `ss.accept()`.
   - For each accepted peer connection, starts another thread that reads the presented code, compares it to `myCode`, and replies `"Connected"` or `"Invalid code"`.

**Connect button handler (`jButton4ActionPerformed`):**

1. Calls `Client.find(...)` to get `host:port:code`.
2. Splits the result on `:` to extract the three parts.
3. If exactly three parts are present, calls `Client.connectToPeer(host, port, code)` and prints the peer's response.
4. If the result is `"Not found"` (fewer than three parts), prints `"Cannot connect: Not found"`.

## 5. Conclusion

### 5.1 Summary

The Finder System successfully meets all stated requirements:

- The Finder Server maintains a registry of clients (name, host, port, unique code).
- Clients can register, unregister, and search for peers.
- The server assigns a unique code per registration and returns it as part of every `FIND` response.
- A registered client starts a peer listener that validates incoming connection codes.
- The **Connect** button automates the full two-step lookup-then-connect flow from a single click.
- Both the Server and Client have functional GUIs that display all activity in real time.

### 5.2 Advantages

| Advantage | Detail |
|---|---|
| **Simplicity** | Only 4 Java source files, no external libraries |
| **Lightweight protocol** | Plain UTF text commands over raw TCP; easy to debug |
| **Unique code security** | A client cannot bypass the Finder – it must look up a target to get the code needed to connect |
| **Concurrent connections** | A new thread per connection means many clients can communicate simultaneously |
| **Clear separation** | GUI classes are separate from logic classes (`Client` / `FinderServer`) |

### 5.3 Disadvantages

| Disadvantage | Detail |
|---|---|
| **No persistence** | The client registry is held in memory; restarting the server wipes all registrations |
| **No authentication** | Any client can register under any name; there is no identity verification |
| **Code is predictable** | Codes are sequential integers starting at 1000; a determined attacker could guess them |
| **Single server** | There is no fault tolerance; if the Finder Server crashes, discovery stops |
| **No encryption** | All traffic (including codes) is sent in plaintext |
| **Peer listener never closes** | The `ServerSocket` opened on registration has no clean shutdown path |

#### `FinderServer.java`

```java
package ServerSide;

import java.net.*;
import java.io.*;
import java.util.ArrayList;
import java.util.concurrent.atomic.AtomicInteger;

public class FinderServer implements Runnable
{
    Socket cSocket;
    static ArrayList<String[]> clients = new ArrayList<String[]>();
    // each entry: {name, host, port, code}
    static AtomicInteger codeCounter = new AtomicInteger(1000);

    public FinderServer(Socket cSocket) { this.cSocket = cSocket; }

    @Override
    public void run() {
        try {
            DataInputStream input   = new DataInputStream(cSocket.getInputStream());
            DataOutputStream output = new DataOutputStream(cSocket.getOutputStream());
            String command = input.readUTF();

            if (command.startsWith("REGISTER")) {
                String[] p = command.split(" ");
                String name = p[1], host = p[2], port = p[3];
                boolean exists = false;
                for (String[] c : clients) if (c[0].equals(name)) { exists = true; break; }
                if (exists) {
                    output.writeUTF("Already registered: " + name);
                } else {
                    String code = String.valueOf(codeCounter.getAndIncrement());
                    clients.add(new String[]{name, host, port, code});
                    output.writeUTF("Registered: " + name + " Code: " + code);
                    FinderServerForm.log("Registered: " + name + " at " + host + ":" + port + " code=" + code);
                }
            } else if (command.startsWith("FIND")) {
                String name = command.split(" ")[1];
                String result = "Not found";
                for (String[] c : clients)
                    if (c[0].equals(name)) { result = c[1] + ":" + c[2] + ":" + c[3]; break; }
                output.writeUTF(result);
                FinderServerForm.log("Find: " + name + " -> " + result);
            } else if (command.startsWith("UNREGISTER")) {
                String name = command.split(" ")[1];
                boolean removed = false;
                for (int i = 0; i < clients.size(); i++)
                    if (clients.get(i)[0].equals(name)) { clients.remove(i); removed = true; break; }
                if (removed) { output.writeUTF("Unregistered: " + name);   FinderServerForm.log("Unregistered: " + name); }
                else         { output.writeUTF("Not found: " + name); }
            }

            input.close(); output.close(); cSocket.close();
        } catch (Exception e) { System.out.println(e); }
    }
}
```

#### `Client.java`

```java
package ClientSide;

import java.net.*; import java.io.*;

public class Client {
    public static String register(String host, int port, String name, String myPort) throws IOException {
        Socket s1 = new Socket(host, port);
        DataOutputStream out   = new DataOutputStream(s1.getOutputStream());
        DataInputStream  input = new DataInputStream(s1.getInputStream());
        out.writeUTF("REGISTER " + name + " localhost " + myPort);
        String result = input.readUTF();
        input.close(); out.close(); s1.close();
        return result;
    }

    public static String unregister(String host, int port, String name) throws IOException {
        Socket s1 = new Socket(host, port);
        DataOutputStream out   = new DataOutputStream(s1.getOutputStream());
        DataInputStream  input = new DataInputStream(s1.getInputStream());
        out.writeUTF("UNREGISTER " + name);
        String result = input.readUTF();
        input.close(); out.close(); s1.close();
        return result;
    }

    public static String find(String host, int port, String name) throws IOException {
        Socket s1 = new Socket(host, port);
        DataOutputStream out   = new DataOutputStream(s1.getOutputStream());
        DataInputStream  input = new DataInputStream(s1.getInputStream());
        out.writeUTF("FIND " + name);
        String result = input.readUTF();
        input.close(); out.close(); s1.close();
        return result;
    }

    public static String connectToPeer(String host, int port, String code) throws IOException {
        Socket s1 = new Socket(host, port);
        DataOutputStream out   = new DataOutputStream(s1.getOutputStream());
        DataInputStream  input = new DataInputStream(s1.getInputStream());
        out.writeUTF(code);
        String result = input.readUTF();
        input.close(); out.close(); s1.close();
        return result;
    }
}
```

### D. Hardships Faced

1. **Thread-safe code generation** – Using a simple `int++` counter across multiple server threads would cause race conditions. This was solved by using `java.util.concurrent.atomic.AtomicInteger`, which guarantees that each `getAndIncrement()` call returns a unique value even when called from several threads simultaneously.

2. **Keeping the GUI class free of extra classes** – The peer listener had to run as a background thread inside `ClientForm` without adding a new `PeerListener` class. This was solved by nesting two anonymous `Runnable` instances (one for the accept loop, one per peer connection) directly inside the Register button handler.

3. **Parsing the server response for the code** – The Register response contains the code embedded in a human-readable string (`"Registered: Alice Code: 1000"`). A simple `split("Code: ")[1].trim()` was sufficient to extract just the numeric code.

4. **Preventing double-listening** – If a user clicks Register twice, the application should not try to open the same port twice. A boolean `listening` flag was added as an instance variable to prevent starting a second `ServerSocket`.
